js备忘笔记| 6.2 创建对象

发表于 2018-03-23 | 分类于 JavaScript ， 笔记

工厂模式

特点：不使用 new 操作符，无法判断对象的类型

构造函数模式

特点：

  • 没有显示的创建对象
  • 直接将属性和方法赋值给 this 对象
  • 没有 return 语
  • 可以通过 instanceof 操作符检测对象类型

new 的四个过程

  • 创建一个新对象
  • 将构造函数额作用域赋值给对象，因此 this 指向这个对象
  • 执行构造函数中的代码
  • 返回新的对象

存在的问题 ：每个方法都在每个实例上重新创建一遍。不同实例上的同名函数是不相等的。

原型模式

prototype ：每个函数都有一个 prototype 属性，它是一个指针，指向一个对象。 可以包含特定类型的所有实例的共享的方法和属性

constructor ：指向拥有该原型的构造函数

[[Prototype]] 指向原型对象，私有属性

  • 当对象实例添加一个属性时，这个属性就会屏蔽原型对象中保存的同名属性

  • hasOwnPrototype() 检测一个属性是否存在于实例中

原型与 in 操作符

in 会在通过对象能访问的属性是返回 true ,不管属性是位于实例中还是原型中

for-in 循环会返回对象的可访问属性，IE8之前的版本不支持

ps ：虽然我们可以通过对象实例访问保存在原型中的值，但却不能访问通过对象实例重写原
型中的值。（但是引用就不同了）

字面量的方式创建原型

此时的constructor 是指向Object的，因为此时实际上是创建了一个新的对象，其实是调用了 Object 构造函数，也就是说constructor 实际上是Object的。但是此时使用instanceof 操作符依旧能检测出为 Object 和 Person 类型。

可以在Person的原型中将 constructor 强行指回 Person 中。 但是这样 constructor 就变成可以枚举的属性了，默认情况下是不可以的。

ps：重写原型以后会切断已经存在的实例与原型之前的关系。

原型对象存在的问题

  1. 省略了为构造函数传参的环节
  2. 包含引用类型属性时，实例上的修改都将影响到原型上，也就将影响所有的实例（但是实例上基本类型的修改不影响原型上的）

构造函数+原型模式 (组合模式，较常用)

将不共享的属性放在构造函数中，将共享的方法放在原型中。这样就可以给构造函数传参，实现方法共享，又解决引用类型属性共享的问题。

动态原型模式 (封装的思想)

寄生构造函数（不推荐）

应用场景：为特殊情况下的对象创建构造函数。

eg：假设想创建一个具有额外方法的特殊数组，但是又不能修改Array 构造函数时。

存在的问题：返回的对象与构造函数或者构造函数的原型属性之间没有关系，不能通过 instanceof 来检测对象的类型。

委托构造函数模式 （适用于一些安全环境中）

有些安全环境中禁止使用 new 和 this ，或者在防止数据被其他应用程序改动时。

同样的，它不可以通过 instanceof 检测对象的类型

与寄生模式的不同之处 ：

  • 创建对象的实例方法不用 this
  • 不使用 new 操作符
